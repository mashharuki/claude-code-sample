# ピッチプレイブック — デモ動画と Devpost 提出文

情報源: RevenueCat 公式「How to win Shipaton part 4: pitching your app」

## 大原則

**審査員が見るのはデモ動画と説明文だけ。** 何十本もの提出を見る審査員が覚えているのは「ストーリー」であって機能リストではない。プロダクトが90点でもピッチが40点なら勝てない。

準備は最終週ではなく**9月中旬に開始**する。動画の撮り直し・編集には想像の3倍かかる。

## デモ動画（最大2分）

### 形式要件
- 最大2分（審査員はこの長さまでしか見る義務がない）※2025年は3分だった。最新ルールを確認
- 実機（ターゲットデバイス）で動作している様子を必ず含める
- YouTube / Vimeo 等の公開プラットフォームにアップロード
- 著作権保護された音楽・素材の無断使用は禁止

### 構成: Story Circle（Dan Harmon の8ステップ）
1. **Setup**: 共感できる人物・状況（あなた自身でよい）
2. **Need**: 課題。**最初の15〜20秒で「何の課題を・誰のために」を伝えて審査員を掴む**
3. **Go**: アプリの登場（ソリューション）
4. **Search**: 機能を実際に使う様子
5. **Find**: 勢いが出てくる（うまくいく瞬間）
6. **Take**: 努力と報酬（トラクション・数値を見せるならここ）
7. **Return**: 新しい日常
8. **Change**: 変化・変革（課題が解決された世界）

### よくある失敗
- 機能チェックリストの読み上げになっている
- ユーザー価値でなく UI の細部説明に終始している
- ペースが遅い（冒頭にロゴアニメ10秒、は自殺行為）

## Devpost 提出文

### 冒頭: ロジックライン1行
> 「**[誰]** のための **[アプリ名]** は、**[独自のアプローチ]** によって **[片付けたいジョブ]** を解決し、**[価値ある結果]** をもたらす」

### 続く構成
1. 課題の段落（誰が・どんな痛みを・なぜ今）
2. ソリューションの段落（アプリがどう解決するか）
3. **トラクション・ユーザー検証の証拠**（DL数、課金ユーザー数、レビュー引用、Build in Public の反響）
4. その後に機能リスト・技術スタック・RevenueCat 統合の説明（HAMM 狙いなら課金設計の思想をここで語る）

## 提出前の最終チェックリスト

- [ ] デモ動画: 2分以内・実機映像・公開URL・冒頭15秒で課題が伝わる
- [ ] 説明文: ロジックラインで始まり、トラクションの証拠がある
- [ ] 公開ストア URL が有効（審査期間中に公開が取り下げられないこと）
- [ ] アイコン 1024×1024 / スクリーンショット 1179×2556（フレームなし）
- [ ] 審査員向けプロモコード or フリートライアルが機能する（実際に新規アカウントで確認）
- [ ] 狙いカテゴリを提出フォームで正しく選択
- [ ] #BuildInPublic 狙いなら発信の証跡（ポストへのリンク集）を添付
- [ ] 締切は 9/30 23:45 PDT = **日本時間 10/1 15:45**。ただしストア審査リードタイムがあるため、アプリ自体はもっと早く公開しておく
