# 実行プレイブック — 技術選定・4-8-24法・グロース

情報源: RevenueCat 公式「How to win Shipaton part 2: building fast」「part 3: growing your app」

## 技術スタック選定の判断基準

原則: **一番速く出荷できるのは、一番慣れているスタック**。ハッカソン中に新フレームワークを学ばない。

| 状況 | 推奨 |
|---|---|
| React/JS 経験者 | React Native + Expo（EAS でビルド・提出まで最速。RevenueCat SDK 対応良好） |
| iOS/Android 両取りしたい・モバイル未経験 | Flutter（UIの磨き込みがしやすく Design Award と相性が良い） |
| Kotlin 経験者 | Kotlin Multiplatform（2025年は専用賞もあった。Momental が実証） |
| iOS のみ・Swift 経験者 | SwiftUI ネイティブ（Apple プラットフォームの完成度で勝負） |

アーキテクチャの原則:
- **バックエンドレスを基本**とする。ローカルストレージ＋（必要なら）Firebase/Supabase 等の BaaS。自前サーバーは2ヶ月の敵
- AI 機能は API 直叩き（Claude API / OpenAI / Whisper）。API コストは価格設計に織り込む
- 課金は RevenueCat に全部任せる: Products / Offerings / Entitlements + RevenueCat Paywalls（ペイウォールUIをノーコードで A/B テスト可能）

## RevenueCat 統合の設計チェックリスト

- [ ] Entitlement は「機能へのアクセス権」で命名する（例: `pro`）。商品IDと分離する
- [ ] Offering で価格プランをグルーピングし、ダッシュボードから差し替え可能にする
- [ ] ペイウォール表示タイミングを設計する（オンボーディング直後 / コア価値体験直後 / 機能ゲート時）
- [ ] 審査員用に**フリートライアルまたはプロモコード**を必ず用意する（提出要件）
- [ ] サンドボックステスト（iOS: Sandbox Apple ID / Android: License Testing）を最初の週に通す
- [ ] HAMM 狙いなら: 価格の思想（なぜこの価格か）、ミッション連動（課金が誰かに貢献する設計）、トライアル→転換の導線を文書化する

実装の詳細は RevenueCat 公式ドキュメント（docs、Codelabs）と、利用可能なら RevenueCat 公式スキル群（`RevenueCat:integrate-revenuecat`、`RevenueCat:revenuecat-paywall` 等）に従う。

## 4-8-24 法（公式推奨の開発フレーム）

「頭の中のアイディア」→「ユーザーの手の中のアプリ」をスコープクリープなしで進める型。

### 4時間ビルド: コンセプト実証
- 動くことを証明する最小プロトタイプ
- **Wizard of Oz 法**を使ってよい: UI は本物、裏側は手動・モック。「レスポンスが本物であるかのように作る」
- ここで「大量の追加作業が必要」と判明したアイディアは殺す

### 8時間ビルド: ユーザーテスト可能版
- 主要機能＋磨かれた UI＋「デモに耐える最低限の内部実装」
- 「見た目は上質、粗さは隠れている、説得力のあるフェイク」でよい。プロダクションコードにしない
- この版を実ユーザーに触らせて検証する（ideation-playbook の検証ループへ）

### 24時間ビルド: 磨き込み（8時間×3日）
- MLP の80%は存在する前提で、残り20%の磨き込みに充てる
- ストアページ作成（スクショ・説明文・キーワード）、ローンチ素材の準備もここ

### その後のリズム
- **週次リリース**: 初期は新機能よりバグ修正を優先。定期的な磨き込みがチャーンを減らす
- 早期テスター（TestFlight / Internal Testing）で壊れた機能と分かりにくい UX を洗い出す
- AI 活用: UI スキャフォールド・ボイラープレート・マーケコピーは AI に書かせ、出力は「下書き」として人間が仕上げる

## グロース: 課金ユーザー100人への道

公式の目標設定: **Shipaton 期間中に課金顧客100人**。野心的だが現実的で、グロースハック不要。

### ユーザー獲得（順番に広げる）
1. **既存ネットワーク**: 友人・家族・その課題を持つ知人。LinkedIn/X で「何の課題を解決するか」を投稿
2. **ニッチコミュニティ**: Reddit・Discord・専門フォーラムでターゲットが集まる場所を探す（宣伝ではなく課題の会話に参加する）
3. **マイクロインフルエンサー**: アプリの無料アクセスと引き換えに紹介してもらう
4. **Shipaton 自体を成長装置に使う**: 公式 Discord（#lang-jp チャンネルあり）でアプリを見せ、参加者同士でクロスプロモーション、IRL イベント参加
5. 初期ユーザーを特別扱いする: 早期割引、ファウンディングユーザーバッジ、イースターエッグ

### ASO（ストア最適化）
- 実際に検索されるキーワードをタイトル・サブタイトル・説明文に入れる
- 説明文は機能列挙でなく**ユーザーが得るベネフィット**を先に
- スクリーンショットは「1枚目で価値が伝わる」構成に
- レビューは直接依頼する（「正直なフィードバックが発見可能性を助ける」と説明する）。リリースノートは「バグ修正」でなく具体的に書く

### リテンション＆フィードバックループ
- 獲得よりも維持。フィードバックとバグに即応し、修正したらユーザーに通知する
- 初期ユーザーには DM・メールで個別にお礼を伝える
- アプリ内で軽量な質問（「この機能は役に立ちましたか？」）、濃い対話は専用 Discord/Slack で
- コミュニティ駆動のアプリはマネタイズ（転換率）も強い傾向がある

### Build in Public の運用（#BuildInPublic Award 対策兼グロース）
- 8/1 から週2〜3回、X（+必要に応じて LinkedIn/TikTok）で発信
- 投稿すべき内容: 数値（DL・売上・転換率）の透明な共有、失敗と学び、ユーザーフィードバックをどう反映したかのビフォーアフター、開発の舞台裏
- ハッシュタグ #Shipaton / #BuildInPublic を使い、他の参加者とも絡む
- 審査対象は「ストーリーとして魅力的か」。単なる進捗報告でなく、山あり谷ありの物語として通貫させる
